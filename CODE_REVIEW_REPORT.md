# 代码审查报告

- 报告日期：2026-03-06
- 审查范围：仓库内一方应用代码、构建脚本、部署文档与关键静态资源加载逻辑
- 明确未逐行审查：`js/vendor/*.js` 与 `js/china.js` / `js/province/*.js` 的三方/生成地图数据；未读取 `js/data.js` 的实际学生内容
- 总体结论：`REQUEST_CHANGES`

## 审查摘要

这个仓库的主要风险不在 UI，而在数据暴露模型和失败处理模型：

1. 当前部署方式会把学生数据作为静态资源直接下发到浏览器，和“通过环境变量安全存储”的描述不一致。
2. 数据输入缺少结构与名称校验，只要有一条坏数据，地图初始化就可能直接崩溃。
3. 资源加载失败时页面容易进入“空白但不报错”的状态，现场排查成本高。

## Findings

### P0 - Critical

1. 生产环境学生数据实际上仍然对公网可见
   - 位置：
     - `build.js:4-11`
     - `index.html:228-229`
     - `README.md:16-22`
     - `DEPLOYMENT.md:40-59`
   - 问题：
     `STUDENTS_DATA` 只是在构建阶段被写入 `js/data.js`，随后 `index.html` 会把这个文件作为普通静态脚本公开加载。只要站点是公网可访问的，任何访问者都能直接拿到完整学生数据。环境变量只隐藏了源码仓库和 CI 配置，不隐藏最终交付物。
   - 影响：
     这与 README 中“安全存储”的表述相冲突，也和“仅供内部使用”的目标不一致；如果数据包含真实姓名、学校、城市、省份，这已经是直接的隐私暴露面。
   - 建议修复：
     - 如果数据必须保密，不要继续用纯静态站点把完整数据发到前端。
     - 至少改为受访问控制的交付方式，例如 Cloudflare Access、身份认证网关，或由受保护的后端按权限返回最小化数据。
     - 立即修正文档，删除“环境变量即可安全存储生产数据”的表述，避免给维护者错误安全预期。

### P1 - High

2. 数据没有做 schema 和名称校验，单条坏记录即可让地图初始化崩溃
   - 位置：
     - `build.js:7-11`
     - `js/map.js:61-64`
     - `js/map.js:77-81`
   - 问题：
     构建脚本只做 `JSON.parse`，不验证 `STUDENTS_DATA` 是否为数组，也不验证每条记录是否包含合法的 `province` / `city`。运行时却直接访问 `provinces[s.province].value`。如果出现 `北京市` 与 `北京` 不一致、尾部空格、空字符串、对象结构错误等情况，页面会在初始化阶段抛异常。
   - 影响：
     这不是“数据少一条”的降级，而是整页地图可能完全不可用；而当前加载器又没有把这个错误清晰反馈给用户。
   - 建议修复：
     - 在 `build.js` 中校验：顶层必须是数组；每项必须是对象；`name/school/city/province` 必须是非空字符串。
     - 在构建阶段校验 `province` 是否命中地图省份字典，必要时做规范化映射。
     - 运行时对异常记录做跳过和告警，不要让单条坏数据阻断整页渲染。

### P2 - Medium

3. 关键脚本加载失败时会被静默吞掉，页面容易变成“空白成功态”
   - 位置：
     - `index.html:187-190`
     - `index.html:232-245`
   - 问题：
     `loadNext` 对任意脚本 `onerror` 都只是 `console.error` 后继续加载下一个脚本，最后无论关键脚本是否真正可用，都会隐藏 loading。`js/china.js`、`js/data.js`、`js/map.js` 任何一个缺失或执行失败，用户都可能只看到空地图，没有明确错误界面。
   - 影响：
     线上故障不容易被最终用户感知为“加载失败”，而更像页面本身没有内容；排查会依赖浏览器控制台。
   - 建议修复：
     - 把脚本分成“必须成功”和“可降级”两类。
     - `js/china.js`、`js/data.js`、`js/map.js` 失败时直接显示错误覆盖层，不要继续进入正常完成态。
     - 为初始化过程增加统一错误边界，例如 `window.onerror` / `unhandledrejection` 的兜底上报与 UI 提示。

### P3 - Low

4. 分享卡片的“覆盖城市”统计会把不同省份的同名城市/区县算成一个
   - 位置：
     - `js/map.js:547-561`
   - 问题：
     `citySet.add(s.city)` 只按城市名去重，没有把省份纳入 key。中国不同省份中同名区县很常见，分享图里的“覆盖城市”会被低估。
   - 影响：
     这是分享功能里的轻度数据正确性问题，不影响主地图展示，但会让导出的统计值失真。
   - 建议修复：
     使用复合 key，例如 `${s.province}::${s.city}`。

## Removal / Iteration Plan

### Safe to remove now

1. 未使用的分享模板 DOM 与对应样式
   - 位置：
     - `index.html:159-183`
     - `css/main.css:974-1098`
   - 依据：
     当前分享实现已经改为 Canvas 直接合成，代码中没有再读取 `#share-template`、`#share-map-container`、`#stat-total`、`#stat-provinces`、`#stat-cities`。
   - 价值：
     删除后可减少误导性结构，避免维护者误以为分享仍依赖隐藏 DOM 模板。

2. 与项目无关的示例文件 `js/default.template.js`
   - 位置：
     - `js/default.template.js:1-25`
   - 依据：
     该文件是示例柱状图代码，仓库内没有引用，且与当前中国地图站点无关。
   - 价值：
     删除后可以降低新贡献者的理解噪音。

### Defer with plan

1. “静态文件承载敏感数据”的整体架构
   - 原因：
     这不是单文件清理，而是部署模型调整，涉及访问控制、数据最小化和产品使用边界。
   - 建议顺序：
     1. 先决定站点是否必须公网可访问。
     2. 如果必须内网/受限访问，先加访问控制，再迁移数据读取方式。
     3. 最后同步修改 README / DEPLOYMENT，避免继续传播错误部署方式。

## Additional Suggestions

- 把第三方运行时代码尽量改为本地托管，尤其是地图 JS 与字体资源；当前页面会在含敏感数据的场景下执行外部 CDN 脚本。
- 给 `build.js` 增加最小测试样例，至少覆盖“合法数据”“非法 JSON”“字段缺失”“省份不匹配”四类输入。
- 在 README 中明确区分“从 Git 仓库移除”与“对最终访问者不可见”是两个完全不同的安全目标。

## 覆盖与限制

- 这次是整仓静态审查，不是基于某个 diff 的增量 review。
- 没有启动浏览器做人工交互回归，因此 hover、drilldown、分享按钮的最终视觉行为未做实机确认。
- 为避免暴露真实数据，本次没有打开 `js/data.js` 的内容，只审查了其生成与加载路径。
